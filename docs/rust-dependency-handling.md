# Rust Dependency Handling in Turnkey

This document provides detailed guidance for handling Rust dependencies in Turnkey's Buck2 integration. While most crates work automatically, Rust's build script system means some crates require special handling.

## How Rust Dependencies Work

### The Standard Flow

```
Cargo.lock â†’ rust-deps.toml â†’ rust-deps-cell.nix â†’ .turnkey/rustdeps/
```

1. **Cargo.lock** defines exact versions and dependency graph
2. **rust-deps.toml** adds Nix hashes for each crate (generated by rustdeps-gen)
3. **rust-deps-cell.nix** fetches crates and generates rules.star files
4. **gen-rust-buck.py** parses each crate's Cargo.toml to generate its rules.star file

### What Works Automatically

- **Dependencies**: Resolved from `[dependencies]` in Cargo.toml
- **Features**: Unified across the dependency graph (like Cargo does)
- **Crate renaming**: `package = "real-name"` in dependencies
- **Proc-macros**: Detected from `[lib] proc-macro = true`
- **Edition**: Read from `package.edition` (defaults to 2015)
- **Crate root**: Detected from `[lib] path` or standard locations

### What Requires Manual Handling

Rust crates can have `build.rs` scripts that run during compilation. Since we can't run arbitrary code in Nix's sandbox, we must handle build script outputs manually:

| Build Script Output | Example Crate | Solution |
|---------------------|---------------|----------|
| `cargo:rustc-cfg=...` | serde_json, rustix | `rustcFlagsRegistry` |
| Generated `.rs` files | serde, serde_core | `buildScriptFixups` |
| Compiled native code | ring | `buildScriptFixups` with compilation |
| Environment variables | Various | Generally auto-handled via `CARGO_*` |

## Diagnosing Problems

### Symptom: Undefined cfg Flag

**Error example:**
```
error[E0425]: cannot find value `fast_arithmetic` in this scope
```

**Diagnosis:**
The crate's `build.rs` normally sets this via `cargo:rustc-cfg=fast_arithmetic="64"`.

**Solution:**
Add to `rustcFlagsRegistry`:
```nix
rustcFlagsRegistry = {
  serde_json = ["--cfg" ''fast_arithmetic=\"64\"''];
};
```

**How to find the right flags:**
1. Read the crate's `build.rs` on crates.io or GitHub
2. Look for `println!("cargo:rustc-cfg=...")` calls
3. Determine what values apply for your target (usually x86_64-linux)

### Symptom: Missing Generated File

**Error example:**
```
error[E0432]: unresolved import `crate::private`
  --> serde/src/lib.rs:42:9
   |
42 | pub use private::*;
   |         ^^^^^^^ could not find `private` in the crate root
```

**Diagnosis:**
The crate expects a file in `OUT_DIR` that `build.rs` normally generates.

**Solution:**
Add to `buildScriptFixups`:
```nix
buildScriptFixups = {
  serde = { patchVersion, vendorPath, ... }: ''
    mkdir -p "$out/${vendorPath}/out_dir"
    cat > "$out/${vendorPath}/out_dir/private.rs" << 'EOF'
#[doc(hidden)]
pub mod __private${patchVersion} {
    pub use crate::private::*;
}
EOF
  '';
};
```

**How to find what to generate:**
1. Read the crate's `build.rs`
2. Look for file writes to `OUT_DIR`
3. Replicate the generated content

### Symptom: Linker Error for Native Symbols

**Error example:**
```
error: linking with `cc` failed: exit status: 1
  = note: /usr/bin/ld: undefined reference to `ring_core_0_17_14__OPENSSL_cpuid_setup'
```

**Diagnosis:**
The crate has C/assembly code that `build.rs` compiles into a native library.

**Solution:**
Add a complex fixup that compiles the native code:
```nix
buildScriptFixups = {
  ring = { patchVersion, vendorPath, ... }: ''
    # Set up paths
    RING_SRC="$out/${vendorPath}"
    RING_OUT="$out/${vendorPath}/out_dir"
    mkdir -p "$RING_OUT"

    # Generate symbol prefix headers
    mkdir -p "$RING_OUT/ring_core_generated"
    cat > "$RING_OUT/ring_core_generated/prefix_symbols.h" << 'HEADER'
    // ... symbol definitions ...
    HEADER

    # Compile C files
    for src in crypto/*.c; do
      cc -c -I$RING_SRC/include -I$RING_OUT -o "$RING_OUT/$(basename $src .c).o" "$src"
    done

    # Assemble .S files
    for src in pregenerated/*.S; do
      cc -c -I$RING_SRC/include -I$RING_OUT -o "$RING_OUT/$(basename $src .S).o" "$src"
    done

    # Create static library
    ar rcs "$RING_OUT/libring_core_0_17_${patchVersion}__.a" $RING_OUT/*.o
  '';
};
```

This is complex! Fortunately, `ring` is already in the default registry.

### Symptom: Feature Conflict or Missing Feature

**Error example:**
```
error[E0433]: failed to resolve: could not find `alloc` in `serde`
```

**Diagnosis:**
Feature unification didn't enable a required feature, or conflicting features are enabled.

**Solution:**
Use `rust-features.toml` to override:
```toml
# rust-features.toml
[overrides]
# Complete replacement
syn = ["derive", "parsing", "visit"]

# Additive
serde = { add = ["alloc"] }

# Subtractive
some_crate = { remove = ["incompatible_feature"] }
```

## The Registry System

### Version-Aware Lookup

Both registries support version-specific keys that take precedence:

```nix
rustcFlagsRegistry = {
  # Catch-all for any version of rustix
  rustix = ["--cfg" "libc" "--cfg" "linux_like" "--cfg" "linux_kernel"];

  # Specific version override (takes precedence)
  "rustix@0.38.0" = ["--cfg" "libc" "--cfg" "linux_like"];  # Different flags for older version
};
```

**Resolution order:**
1. `"crate@version"` - Exact versioned key
2. `"crate"` - Catch-all for any version
3. Default registry (if exists)
4. Empty (no flags/fixup)

### Fixup Function Context

Fixup functions receive context about the crate:

```nix
buildScriptFixups = {
  my_crate = { crateName, version, patchVersion, key, vendorPath }: ''
    # crateName: "my_crate"
    # version: "1.2.3"
    # patchVersion: "3" (last component)
    # key: "my_crate@1.2.3"
    # vendorPath: "vendor/my_crate@1.2.3"

    echo "Building fixup for ${crateName} version ${version}"
    mkdir -p "$out/${vendorPath}/out_dir"
  '';
};
```

### Available Variables in Shell Fixups

Simple string fixups can use these shell variables:
- `$out` - Nix output directory
- Standard shell utilities (mkdir, cat, cc, ar, etc.)

For complex fixups, prefer the function form for access to version info.

### Nix Interpolation vs Shell Escaping (Important!)

In Nix multiline strings (`'' ... ''`), there's a critical distinction:

```nix
buildScriptFixups = {
  my_crate = { patchVersion, vendorPath, ... }: ''
    # CORRECT: ${patchVersion} is Nix interpolation
    # The Nix variable patchVersion (e.g., "14") is substituted at Nix eval time
    MY_VAR="${patchVersion}"

    # WRONG: ''${patchVersion} escapes the $ for shell
    # This becomes literal ${patchVersion} in the shell script, which is undefined
    MY_VAR="''${patchVersion}"  # Results in empty string!

    # CORRECT: ''$out escapes for shell variable
    # $out is a shell variable set by Nix's runCommand
    echo "Output: $out"
  '';
};
```

**Rule of thumb:**
- Use `${var}` for Nix variables (function arguments like `patchVersion`, `vendorPath`)
- Use `$var` or `''$var` for shell variables (like `$out`, loop variables)

## Lessons Learned from Specific Crates

### serde / serde_core

**Problem:** Generates versioned module aliases in `OUT_DIR/private.rs`.

**Why:** Allows multiple versions of serde to coexist by namespacing internal modules with version number.

**Lesson:** When a crate uses versioned internal modules, include the version in generated code:
```rust
pub mod __private${patchVersion} {
    pub use crate::private::*;
}
```

### serde_json

**Problem:** Build script probes target architecture for fast arithmetic.

**Why:** Uses 64-bit arithmetic on 64-bit targets, 32-bit on 32-bit targets.

**Lesson:** Platform-probing build scripts need target-specific flags. For cross-compilation, you'd need different flags per target.

### rustix

**Problem:** Build script selects syscall backend and platform features.

**Why:** Can use raw Linux syscalls (`linux_raw`) or libc wrappers (`libc`). Different features enable different syscall availability.

**Lesson:** Some crates have multiple "backends" selected at build time. The `libc` backend is more portable but may be slower.

### ring

**Problem:** Compiles ~15 C files and ~14 assembly files into a static library with versioned symbol prefixes.

**Why:** Cryptographic primitives implemented in C/assembly for performance. Symbol prefixing allows multiple versions to coexist.

**Lessons:**
1. Native code compilation requires build tools in Nix derivation
2. Symbol prefixing follows specific patterns from `build.rs`
3. Assembly files need different include paths than C files
4. The output library name must match what Rust code expects

**Key insight:** Ring's complexity comes from:
- Versioned symbol prefixes (e.g., `ring_core_0_17_14__`)
- Header generation for symbol renaming
- Both C compilation and assembly
- Platform-specific file selection (we use x86_64-linux ELF)

## Best Practices

### 1. Check build.rs First

Before adding a new Rust dependency that fails to build:
1. Go to the crate on crates.io
2. Click "Repository" to view source
3. Read `build.rs` to understand what it does
4. Determine if it needs flags, file generation, or native compilation

### 2. Start Simple

Try the simplest fix first:
1. First, check if feature unification is the issue
2. Then try adding to `rustcFlagsRegistry`
3. Then try a simple `buildScriptFixups` string
4. Only write complex fixups if needed

### 3. Version Your Fixups When Necessary

If a crate's build script changes between versions:
```nix
buildScriptFixups = {
  # Old version had different generated content
  "my_crate@1.0.0" = ''...'';

  # New version catch-all
  my_crate = ''...'';
};
```

### 4. Document Complex Fixups

For fixups longer than 10 lines, add comments explaining:
- What the original build.rs does
- Why each step is necessary
- What output format the Rust code expects

### 5. Test After Adding Fixups

After adding a fixup:
```bash
# Regenerate the cell
direnv reload

# Build something that uses the crate
buck2 build //path/to:target

# Verify the output exists
ls -la .turnkey/rustdeps/vendor/crate@version/out_dir/
```

## Debugging Tips

### Inspect Generated rules.star Files

```bash
cat .turnkey/rustdeps/vendor/serde_json@1.0.140/rules.star
```

Look for:
- `rustc_flags` - Should include cfg flags
- `env` - Should include `OUT_DIR` if fixup crate
- `deps` - Dependencies resolved correctly

### Check Fixup Output

```bash
ls -la .turnkey/rustdeps/vendor/ring@0.17.14/out_dir/
```

Should contain generated files/libraries.

### Trace Feature Resolution

The unified features are computed by `compute-unified-features.py`:
```bash
# The features JSON is passed to gen-rust-buck.py
# Check what features a crate gets:
grep -A20 "rust_library" .turnkey/rustdeps/vendor/serde@*/rules.star | grep features
```

### Check for Missing Crates

If dependency resolution fails:
```bash
# See what crates are available
ls .turnkey/rustdeps/vendor/
```

The crate might be missing from rust-deps.toml or have a different name.

## Contributing New Default Fixups

If you encounter a common crate that needs fixups:

1. **Test your fixup thoroughly**
2. **Keep it minimal** - Only do what's necessary
3. **Document why** - Future maintainers need to understand
4. **Submit to the defaults** - Add to `nix/buck2/rust-deps-cell.nix`

The default registries in `rust-deps-cell.nix` help everyone, so contributions for common crates are welcome.
