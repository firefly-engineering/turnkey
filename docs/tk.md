# tk Command Reference

`tk` is the turnkey CLI wrapper for buck2 that automatically keeps generated files in sync.

## Overview

When using Buck2 with Nix-managed dependencies, certain files must be regenerated when source files change:
- `go-deps.toml` must be updated when `go.mod` or `go.sum` changes
- BUCK files may need regeneration based on your sync configuration

`tk` solves this by automatically running sync operations before buck2 commands that read the build graph. This ensures you never build with stale generated files.

## Quick Start

```bash
# Use tk just like buck2 - it syncs automatically
tk build //some:target     # syncs first, then builds
tk test //some:target      # syncs first, then tests
tk run //some:target       # syncs first, then runs

# Explicit sync operations
tk sync                    # manually sync all stale files
tk check                   # check if files are stale (for CI)

# Skip sync when needed
tk --no-sync build //...   # skip sync, run buck2 directly
```

## Command Reference

### tk build/run/test/... (buck2 passthrough)

Most tk commands are passed through to buck2. Commands that read the build graph automatically sync first:

**Sync-first commands** (sync before running):
- `build` - Build targets
- `run` - Run a target
- `test` - Run tests
- `query` - Query the build graph
- `cquery` - Configured query
- `uquery` - Unconfigured query
- `targets` - List targets
- `audit` - Audit the build
- `bxl` - Run BXL scripts

**Pass-through commands** (no sync):
- `clean` - Clean build artifacts
- `kill` - Kill buck2 daemon
- `killall` - Kill all buck2 processes
- `status` - Show daemon status
- `log` - View build logs
- `rage` - Generate debug report
- `help` - Show help
- `docs` - Open documentation
- `init` - Initialize a project

Unknown commands default to syncing first (safe default).

### tk sync

Explicitly synchronize all stale files.

```bash
tk sync              # sync stale files
tk sync --verbose    # show what's being synced
tk sync --dry-run    # show what would be synced without doing it
```

**Exit codes:**
- `0` - Success (files synced or nothing to sync)
- `1` - Sync failed

**Output:**
- Silent when nothing is stale
- Reports number of files synced when files are regenerated
- Shows errors if sync fails

### tk completion

Generate shell completion scripts for bash, zsh, or fish.

```bash
tk completion bash    # output bash completion script
tk completion zsh     # output zsh completion script
tk completion fish    # output fish completion script
```

**Enable completions:**

```bash
# Bash (add to ~/.bashrc)
eval "$(tk completion bash)"

# Zsh (add to ~/.zshrc)
eval "$(tk completion zsh)"

# Fish (run once)
tk completion fish > ~/.config/fish/completions/tk.fish
```

The completion scripts are based on buck2's completions with tk-specific additions for `sync`, `check`, `--no-sync`, and `--dry-run`.

### tk check

Check if any files are stale without regenerating them. Useful for CI validation.

```bash
tk check             # check staleness
tk check --verbose   # show detailed status
```

**Exit codes:**
- `0` - All files up-to-date
- `1` - Files are stale (run `tk sync` to fix)

**Example CI usage:**
```yaml
- name: Check files in sync
  run: tk check
```

## tk Flags

Flags must come before the subcommand:

| Flag | Description |
|------|-------------|
| `--no-sync` | Skip sync, run buck2 directly |
| `--verbose`, `-v` | Show what tk is doing |
| `--dry-run`, `-n` | Show what would be synced without doing it |
| `--help`, `-h` | Show help |

**Examples:**
```bash
tk --no-sync build //...     # skip sync
tk --verbose sync            # verbose sync
tk -v -n sync                # dry-run with verbose output
```

## Configuration

### Sync Configuration File

tk reads staleness rules from `.turnkey/sync.toml`. **This file is automatically generated from your Nix configuration** - you don't need to create or maintain it manually.

When you configure dependency files in your `flake.nix`:
- `goDepsFile` → generates a Go deps rule
- `rustDepsFile` → generates a Rust deps rule
- `pythonDepsFile` → generates a Python deps rule

**Example generated sync.toml:**
```toml
# Generated by turnkey - do not edit manually
[[deps]]
name = "go"
sources = ["go.mod", "go.sum"]
target = "go-deps.toml"
generator = ["godeps-gen", "--go-mod", "go.mod", "--go-sum", "go.sum", "--prefetch"]

[[deps]]
name = "rust"
sources = ["Cargo.toml", "Cargo.lock"]
target = "rust-deps.toml"
generator = ["rustdeps-gen", "--cargo-lock", "Cargo.lock"]
```

Each `[[deps]]` entry defines:
- `name` - Human-readable name for this rule
- `sources` - Files that trigger regeneration when modified
- `target` - The generated file
- `generator` - Command to regenerate the target

The target is considered stale if any source file is newer than the target.

### Customizing Source Paths

If your project uses non-standard paths, configure them in your `flake.nix`:

```nix
turnkey.toolchains.buck2 = {
  enable = true;
  goDepsFile = ./go-deps.toml;
  goModFile = "src/go.mod";      # Custom go.mod location
  goSumFile = "src/go.sum";      # Custom go.sum location

  rustDepsFile = ./rust-deps.toml;
  cargoTomlFile = "rust/Cargo.toml";
  cargoLockFile = "rust/Cargo.lock";

  pythonDepsFile = ./python-deps.toml;
  pyprojectFile = "python/pyproject.toml";
  pylockFile = "python/pylock.toml";  # Optional lock file
};
```

### Shell Integration

When using turnkey's devenv module, these features are enabled by default:

**buck2 alias to tk:**
```bash
# In devenv shell, buck2 is aliased to tk
buck2 build //...   # actually runs: tk build //...
```

Disable with:
```bash
TURNKEY_NO_ALIAS=1 buck2 build //...   # uses raw buck2
```

Or in Nix configuration:
```nix
turnkey.buck2.tk.aliasBuck2 = false;
```

**Automatic sync on shell entry:**
```nix
turnkey.buck2.tk.syncOnShellEntry = true;  # default
```

**Pre-commit hook:**
```nix
turnkey.buck2.tk.preCommitCheck = true;  # default
```

This adds a pre-commit hook that runs `tk check` before commits, preventing commits when files are out of sync.

## Staleness Detection

### How It Works

1. tk reads `.turnkey/sync.toml` for staleness rules
2. For each rule, it compares modification times:
   - If target doesn't exist: stale
   - If any source is newer than target: stale
   - Otherwise: up-to-date
3. Stale files are regenerated using their configured generator

### What Triggers Regeneration

| Change | Effect |
|--------|--------|
| Edit `go.mod` | `go-deps.toml` becomes stale |
| Edit `go.sum` | `go-deps.toml` becomes stale |
| Delete `go-deps.toml` | Regenerated on next sync |
| Add new deps rule | New target tracked for staleness |

### Staleness Checking is Fast

Staleness detection only checks file modification times - no file content is read. This makes `tk sync` fast even when nothing needs to be regenerated.

## Troubleshooting

### Common Issues

**"tk: .buckconfig not found"**

tk looks for `.buckconfig` to find the project root. Make sure you're in a Buck2 project directory or a subdirectory of one.

**"tk: failed to load sync config"**

The `.turnkey/sync.toml` file is missing or invalid. Create it with your staleness rules.

**"tk: sync failed: generator command failed"**

The generator command for a stale file failed. Check that:
- The generator command is correct
- Required tools are in PATH
- Source files exist

**Sync is slow**

If sync takes a long time, the generator commands might be slow. Consider:
- Using `--prefetch` with godeps-gen to cache downloads
- Checking if generators are doing unnecessary work

### Debug Flags

Use verbose mode to see what tk is doing:

```bash
tk --verbose sync         # see what's being synced
tk --verbose build //...  # see sync output before build
tk --dry-run sync         # see what would be synced
```

### Bypass tk

If you need to use raw buck2:

```bash
# Option 1: --no-sync flag
tk --no-sync build //...

# Option 2: TURNKEY_NO_ALIAS environment variable (if using alias)
TURNKEY_NO_ALIAS=1 buck2 build //...

# Option 3: Full path to buck2
/nix/store/.../bin/buck2 build //...
```

## Examples

### Typical Development Workflow

```bash
# Enter devenv shell (auto-syncs)
nix develop

# Work normally - tk handles sync automatically
buck2 build //...
buck2 test //...

# After adding a Go dependency
go get github.com/some/package
tk sync  # regenerate go-deps.toml
buck2 build //...
```

### CI Workflow

```yaml
jobs:
  build:
    steps:
      - uses: actions/checkout@v4
      - uses: DeterminateSystems/nix-installer-action@main

      - name: Check files in sync
        run: nix develop --command tk check

      - name: Build
        run: nix develop --command tk build //...

      - name: Test
        run: nix develop --command tk test //...
```

### Manual Regeneration

```bash
# See what would be synced
tk --dry-run sync

# Sync with verbose output
tk --verbose sync

# Force regeneration by touching source
touch go.mod
tk sync
```
